# План реализации игрового интерфейса Green District

## 0. Принятые стартовые решения

- Цветовая система берется из документа `Цветовая палитра.docx`: теплый светлый management UI, ощущение бумажной карты города, приглушенная пастельная карта и спокойные статусы без neon/pure black/pure white.
- На первом этапе используем простые встроенные/рисуемые значки. Архитектура UI должна предусматривать последующую замену на внешние SVG/PNG ассеты без переписывания карточек и панелей.
- Базовое разрешение проверки: `1152x648`. Дополнительно желательно не ломать `1920x1080`, но основная плотность и читаемость сверяются по `1152x648`.
- `Codex_plan.md` не редактируется в рамках интерфейсной работы. Рабочий план ведется в этом файле.

## 1. Выводы по первичному описанию

Текущий Godot-интерфейс уже закрывает технический MVP: показывает метрики мира, карту районов, списки сущностей, проекты, события и управление скоростью симуляции. Но визуально он все еще ближе к debug-dashboard: районы представлены прямоугольными плитками, метрики и действия конкурируют за внимание, правая область работает как административная таблица, а события почти не ощущаются как часть игрового процесса.

Цель следующего этапа - перейти к полноценному игровому интерфейсу спокойного management/city-sim: карта должна стать главным экраном, HUD - компактным и читаемым, правая панель - контекстной, а события - видимыми через уведомления и маркеры на карте.

## 2. Целевое состояние MVP интерфейса

### Основной экран

- Верхний HUD: название игры, дата/время, скорость, бюджет, население, поддержка, удовлетворенность, настройки/язык.
- Центральная зона: большая 2D top-down карта района, занимающая большую часть экрана.
- Правая контекстная панель: меняется в зависимости от выбранного района, проекта, события или режима.
- Нижняя лента: последние события, предупреждения, завершенные проекты, кризисы.
- Режимы управления: обзор, районы, проекты, события, жители/бизнесы как вторичные вкладки.

### Игровое ощущение

- Игрок должен видеть "живой район", а не набор таблиц.
- Основные показатели должны считываться за 1-2 секунды.
- Каждый район должен иметь визуальный характер: форма, цветовое состояние, дороги, здания, маркеры проблем.
- Действия игрока должны быть представлены карточками проектов с ценой, сроком, влиянием и понятным состоянием запуска.

## 3. Архитектурный подход

### Слои UI

1. `SimulationBridge` остается тонким мостом к `WorldState`, но получает методы для выборок под интерфейс: district summary, project summary, event summary.
2. `MainDashboard` нужно разгрузить: он сейчас одновременно строит весь экран, обновляет данные и создает все карточки.
3. Новые Godot-компоненты:
   - `GameHud` - верхняя панель метрик и времени.
   - `CityMapView` или расширенный `DistrictMapView` - карта, выбор района, маркеры, визуальные слои.
   - `ContextPanel` - правая панель с состояниями selected district/project/event.
   - `ProjectActionPanel` - карточки доступных проектов.
   - `NotificationFeed` - нижняя лента событий.
   - `ThemeFactory` или `UiTheme` - единые цвета, панели, кнопки, шрифты.

### Данные для UI

Для удобства интерфейса стоит добавить легкие view-model классы в Godot-слой, не в ядро симуляции:

- `HudSnapshot`
- `DistrictViewModel`
- `ProjectViewModel`
- `EventViewModel`
- `MapMarkerViewModel`

Они должны собираться из `WorldState`, не становясь частью симуляционной логики.

## 5. Приоритеты MVP

Обязательно для следующей версии:

- новый layout;
- единая тема;
- карта через custom drawing;
- выбор района;
- контекстная панель района;

Можно отложить:

- отдельные экраны статистики;

## 6. Риски

- `MainDashboard` уже слишком много делает. Если продолжать расширять его напрямую, UI станет трудно поддерживать.
- Карта на `Control` с множеством child-node для каждого здания может стать тяжелой. Для MVP лучше рисовать большую часть карты через `_Draw()`.
- Локализация событий пока частично hardcoded, из-за этого русский интерфейс может смешиваться с английскими событиями.
- Нужно не перегрузить первый игровой экран таблицами: citizens/businesses лучше оставить как диагностические вкладки, а не основной UI.

## 9. UI/UX обновление по документу `Обновление интерфейса.docx`

### 9.1. Выводы анализа

Текущий интерфейс уже попал в правильное направление: спокойная management/city-builder стратегия с теплой палитрой, картой в центре, HUD сверху, контекстной панелью справа и событиями как частью игрового процесса. Главное улучшение теперь не в смене стиля, а в усилении визуальной иерархии, читаемости и ощущения живого города.

Сильные стороны, которые нужно сохранить:

- теплая, неагрессивная палитра без neon/glassmorphism;
- центральная роль карты;
- достаточный воздух между блоками;
- спокойное ощущение стратегического интерфейса;
- минималистичные встроенные значки с возможностью последующей замены на ассеты.

Основные проблемы:

- почти все элементы имеют одинаковый визуальный вес;
- HUD не выделяет главные показатели;
- карта читается, но пока выглядит статичной;
- районы и дороги требуют более сильной визуальной сегментации;
- события снизу ощущаются как лог, а не как игровые уведомления;
- правая панель все еще ближе к административной таблице, чем к игровому инспектору;
- не хватает hover/focus feedback и фокусных точек на карте.

### 9.5. Приоритет P2: строительство и инфраструктура

Цель: проекты должны становиться объектами в мире, а не только строками в панели.

- [x] После запуска инфраструктурного проекта создавать визуальный construction marker на карте.
- [x] По завершении проекта превращать marker в постоянный объект района.
- [x] Разделить строительные проекты по типам:
  - дороги;
  - жилье;
  - медицина;
  - полиция;
  - парк;
  - сервисы/бизнес.
- [x] Доработать отображение стоимости и длительности проекта:
  - upfront cost;
  - ongoing cost;
  - expected effect;
  - progress.
- [x] Динамически генерировать дороги как инфраструктурную необходимость для новой застройки.
  - дороги строятся как доступ к объектам, а не как независимая декоративная сетка;
  - здание размещается только при соседстве с дорогой или при возможности построить к нему сухопутный подъезд;
  - если подъезд к зданию невозможен, место считается недоступным и объект там не создается;
  - локальные районные дороги клипуются по границе района и суше;
  - между соседними районами всегда создается минимум одна межрайонная дорога, соединяющая их локальные дорожные сети;
  - участки любых дорог над водой отображаются как мосты отдельным коричневым цветом, только если оба конца сегмента находятся на суше;
  - мост может быть межрайонным или внутренним районным, но не может начинаться/заканчиваться в воде.
- [x] Временно убрать дорожную traffic-анимацию, чтобы дороги не воспринимались как отдельные движущиеся объекты.

### 9.9. Критерии приемки UI/UX обновления

- В `1152x648` игрок сразу видит бюджет, население и поддержку как главные показатели.
- Выбранный район и район под курсором различаются без чтения tooltip.
- Основные дороги читаются на карте с первого взгляда.
- Событие с выбором выглядит как игровая ситуация, а не строка лога.
- Правая панель показывает не только значения, но и проблемы/рекомендации.
- На карте есть хотя бы один визуальный признак жизни или процесса.
- Стиль остается спокойным, бумажно-картографическим, без резких современных эффектов.
