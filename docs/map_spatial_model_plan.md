# План перехода карты к пространственной модели

## 0. Цель

Перевести карту Green District от процедурной отрисовки линий и прямоугольников к логической пространственной модели, на которую позже можно надежно наложить PNG-ассеты, текстурные тайлы дорог, воды, земли, мостов и зданий.

Карта должна стать источником правды для размещения объектов:

- где находится земля, вода, дорога, мост, здание;
- какие клетки заняты;
- какие объекты имеют доступ к дороге;
- где можно построить новый объект;
- как проложить дорожный маршрут;
- какой визуальный тайл нужен для конкретной клетки дороги или поверхности.

## 1. Базовые решения

- Логическая единица карты: `1 клетка = 1 внутригровой квадратный метр`.
- Видимая сетка не обязательна. Сетка нужна как внутренняя модель для расчетов.
- Экранные пиксели не равны метрам напрямую. Рендер использует масштаб и камеру.
- У карты должен быть максимальный логический размер, чтобы симуляция знала предел расширения мира.
- Стартовая игра начинается с одного района; новые районы появляются только по правилам симуляции.
- Все объекты карты должны иметь размеры в метрах: ширина, длина, занимаемый footprint.
- Дороги и мосты являются объектами сетки, а не просто линиями отрисовки.
- Мост может быть внутри района или между районами, но каждый мостовой сегмент обязан начинаться и заканчиваться на суше.
- Здание считается доступным, если одной стороной или частью стороны соседствует с дорогой.
- Если здание не может получить дорожный доступ, оно не должно размещаться в выбранном месте.

## 2. Целевая структура данных

### 2.1. `MapGrid`

- [x] Ввести логическую модель карты `MapGrid`.
- [x] Хранить размеры карты в метрах: `widthMeters`, `heightMeters`.
- [x] Хранить массив клеток `MapCell`.
- [x] Добавить преобразование координат:
  - [x] grid -> world/map position;
  - [x] world/map position -> screen position;
  - [x] screen position -> grid.
- [x] Отделить логическую карту от рендера `DistrictMapView`.
- [x] Ввести максимальный размер одной сетчатой карты:
  - ориентир: площадь около `10` стандартных районов;
  - ограничение хранится в настройках генерации карты;
  - генератор и расширение районов не могут выходить за этот предел.
- [x] Хранить свободные клетки карты отдельно от клеток районов:
  - свободная земля;
  - вода;
  - недоступные/заблокированные зоны;
  - резерв под будущие районы;
  - резерв под межрайонные дороги и логистику.

### 2.2. `MapCell`

- [x] Ввести тип поверхности клетки:
  - `Land`;
  - `Water`;
  - `Park`;
  - `Blocked`.
- [x] Ввести слой инфраструктуры:
  - пусто;
  - дорога;
  - мост;
  - перекресток.
- [x] Ввести слой объекта:
  - пусто;
  - здание;
  - проект строительства;
  - маркер события.
- [x] Добавить флаги занятости:
  - можно строить;
  - можно прокладывать дорогу;
  - можно проложить мост;
  - доступно для пешеходного/дорожного доступа.

### 2.3. `PlacedMapObject`

- [x] Ввести стандартную структуру объекта карты:
  - id;
  - тип объекта;
  - district id;
  - позиция на сетке;
  - ширина в метрах;
  - длина в метрах;
  - rotation;
  - footprint cells;
  - access sides;
  - asset key.
- [x] Связать игровые сущности с объектами карты:
  - [x] business -> placed object;
  - [x] government project -> placed object;
  - [x] housing/service object -> placed object;
  - [x] marker/event -> map marker.
- [x] Добавить стабильную генерацию id/позиции, чтобы карта не прыгала между перерисовками.

### 2.4. Свободное пространство и границы районов

- [x] Ввести модель границ района как набора клеток, а не только прямоугольной области.
- [x] Рассчитывать свободное пространство по краям района:
  - север;
  - восток;
  - юг;
  - запад.
- [x] Разрешать расширение района только в сторону, где есть свободные клетки карты.
- [x] При расширении района обновлять:
  - [x] принадлежность клеток району;
  - [x] доступные строительные места;
  - [x] дорожную сеть;
  - [x] зоны покрытия объектов;
  - [x] визуальную границу района.
- [x] Если свободного пространства рядом с районом нет, расширение должно отклоняться или запускать сценарий создания нового района.

### 2.5. Жизненный цикл районов

- [x] На старте генерации мира создавать только один район.
- [x] Остальные районы должны появляться через правила симуляции, а не как стартовая раскладка.
- [x] Ввести причины образования нового района:
  - [x] нехватка места под производственные объекты, например фермы;
  - [x] рост плотности населения;
  - [x] потребность в новых ресурсах или производственных цепочках;
  - [x] логистическая удаленность объектов;
  - [x] социальное расслоение или элитарность богатых жителей;
  - [x] специализация бизнеса или ресурсов;
  - [x] невозможность расширить существующий район.
- [x] Новый район должен получать стартовую дорожную связь с существующим районом.
- [x] Новый район должен занимать свободное пространство карты и не пересекаться с существующими районами.

## 3. Размеры объектов

### 3.1. Каталог размеров

- [x] Создать каталог стандартных размеров объектов.
- [x] Ввести приблизительные размеры небольших объектов:
  - малый дом: `8 x 10 м`;
  - средний дом: `10 x 14 м`;
  - магазин: `12 x 16 м`;
  - мастерская: `18 x 24 м`;
  - ферма/теплица: `24 x 18 м`;
  - клиника: `20 x 28 м`;
  - школа: `35 x 45 м`;
  - полиция: `18 x 22 м`;
  - парк: переменный footprint;
  - районная дорога: ширина `6-8 м`;
  - магистраль/межрайонная дорога: ширина `8-10 м`.
- [x] Вынести размеры в data/config, чтобы позже менять баланс без переписывания рендера.

### 3.2. Правила footprint

- [x] Проверять весь footprint объекта, а не только центр.
- [x] Запрещать пересечения с:
  - водой, если объект не водный;
  - дорогами;
  - мостами;
  - другими зданиями;
  - границей района;
  - заблокированными клетками.
- [x] Добавить минимальные отступы между объектами.
- [x] Добавить правило ориентации объекта относительно дороги.

## 4. Доступность зданий

- [x] Здание считается доступным, если хотя бы часть одной стороны соседствует с дорогой.
- [x] Соседство должно проверяться по клеткам, а не по расстоянию до линии.
- [x] У здания должны быть разрешенные стороны доступа.
- [x] Если рядом с существующей дорогой есть место, новое здание занимает это место.
- [x] Если рядом с дорогой места нет, здание пробует разместиться в другом допустимом месте района.
- [x] Для нового места строится дорожный маршрут.
- [x] Если маршрут построить нельзя, место отклоняется.
- [x] Если ни одно место не подходит, объект не создается на карте.

### 4.1. Зоны действия объектов

- [x] Для торговых и социальных объектов ввести радиус или зону действия в метрах/клетках.
- [x] Зона действия должна учитывать дорожную доступность, а не только прямое расстояние.
- [x] Рассчитывать покрытие населения товарами и услугами:
  - [x] сколько жителей находится в зоне действия;
  - [x] какой процент района покрыт;
  - [x] какие потребности остаются без покрытия.
- [x] Для объектов разного типа хранить разные параметры покрытия:
  - [x] магазин/торговля;
  - [x] клиника;
  - [x] школа;
  - [x] полиция;
  - [x] парк;
  - [x] ферма/производство как источник ресурсов, а не прямого социального покрытия.
- [x] Использовать зоны действия при принятии решений о новых проектах и новых районах.

## 5. Дорожная система

### 5.1. Типы дорог

- [x] Ввести типы дорожных клеток:
  - `LocalRoad`;
  - `RegionalRoad`;
  - `AccessRoad`;
  - `Bridge`.
- [x] У дороги хранить ширину в метрах.
- [x] У дороги хранить связность по направлениям: N/E/S/W.
- [x] У дороги хранить принадлежность к району или межрайонной сети.
- [x] На старте района не строить пустую дорожную сетку на будущее: первая локальная дорога создается как доступ к стартовому объекту, а последующие дороги расширяются только для подключения размещаемых объектов.

### 5.2. Перекрестки

- [x] Рассчитывать перекрестки по соседним дорожным клеткам.
- [x] Ввести bitmask связности:
  - 1 направление: тупик;
  - 2 противоположных: прямая;
  - 2 смежных: поворот;
  - 3 направления: T-перекресток;
  - 4 направления: перекресток.
- [x] На основе bitmask выбирать будущий PNG-тайл дороги.
- [x] Пересечения дорог должны образовывать перекресток, а не наложение линий.

### 5.3. Межрайонные дороги

- [x] Между соседними районами всегда должна быть минимум одна дорога.
- [x] Межрайонная дорога должна соединять локальную дорожную сеть одного района с локальной сетью другого.
- [x] Если между районами вода, маршрут может использовать мост.
- [x] Мостовой участок допустим только если начало и конец мостового сегмента находятся на суше.
- [x] Межрайонная дорога должна участвовать в общей системе связности и перекрестков.

### 5.4. Мосты

- [x] Мост является дорожным сегментом над водой.
- [x] Мост может быть:
  - внутри района;
  - между районами;
  - частью подъездного маршрута.
- [x] Мост не может начинаться или заканчиваться в воде.
- [x] Мост должен быть соединен с обычной дорогой на суше.
- [x] Мостовые клетки должны иметь отдельный asset key.

## 6. Маршрутизация

- [x] Ввести pathfinding по сетке.
- [x] Использовать A* или аналогичный алгоритм.
- [x] Вес маршрута должен учитывать:
  - [x] длину пути;
  - [x] пересечение воды;
  - [x] стоимость мостов;
  - [x] обход зданий;
  - границы района;
  - [x] предпочтение соединения с существующей дорогой.
- [x] Маршрут дороги должен строиться по клеткам, а не прямой линией.
- [x] При пересечении существующей дороги маршрут должен подключаться к ней и создавать перекресток.
- [x] Если маршрут невозможен, строительство объекта или дороги отклоняется.

## 7. Подготовка к PNG-ассетам и тайлам

### 7.1. Asset keys

- [x] Для каждого объекта ввести `assetKey`.

  - road cross;
  - road end;
  - bridge straight;

- [x] Для поверхностей ввести asset keys:

  - shoreline;
  - park/trees.
- [x] Генерировать естественные водоёмы с неровной береговой линией, а не прямоугольной заливкой.

### 7.2. Рендер

- [x] Перевести `DistrictMapView` на рендер из `MapGrid`.
- [x] Временно сохранить fallback-отрисовку примитивами, если PNG отсутствует.
- [x] Добавить слойность:
  - земля;
  - вода;
  - берега;
  - дороги/мосты;
  - здания;
  - маркеры;
  - UI overlays.
- [x] Подготовить масштабирование ассетов под grid cells.
- [x] Доработать камеру карты:
  - текущий вид не должен быть максимально отдаленным;
  - должна быть возможность отдалить карту для обзора всей доступной сетки;
  - приблизить карту для просмотра объектов;
  - ограничивать pan/zoom по размеру логической карты, а не только по размеру viewport.

## 9. Критерии готовности
